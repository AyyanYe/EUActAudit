# Contributing to EUActAudit

First off, thank you for considering contributing to EUActAudit! It's people like you that make the open-source community such an amazing place to learn, inspire, and create.

We welcome contributions of all kinds: bug fixes, new features, documentation improvements, and feedback.

## Code of Conduct

By participating in this project, you are expected to uphold our Code of Conduct. Please treat all contributors and users with respect, foster a welcoming environment, and keep interactions professional and collaborative.

## Getting Started

If you want to run the project locally to develop new features or test bug fixes, please refer to our comprehensive [Startup Guide](STARTUP_GUIDE.md) for detailed instructions on setting up both the Python backend and the React frontend.

## Branching Strategy

We follow a standard feature-branch workflow. **Direct pushes to the `main` branch are strictly prohibited.**

1. **Fork** the repository on GitHub.
2. **Clone** your fork locally: `git clone https://github.com/YOUR_USERNAME/AI-Auditor.git`
3. **Create a branch** for your feature or bugfix:
   - For features: `git checkout -b feat/your-feature-name`
   - For bug fixes: `git checkout -b fix/your-bug-name`
   - For documentation: `git checkout -b docs/your-doc-update`

## Commit Guidelines

We use [Conventional Commits](https://www.conventionalcommits.org/) to keep our commit history clean and readable. Please format your commit messages accordingly:

- `feat:` (New feature)
- `fix:` (Bug fix)
- `docs:` (Changes to the documentation)
- `style:` (Formatting, missing semi colons, etc; no production code change)
- `refactor:` (Refactoring production code, e.g. renaming a variable)
- `test:` (Adding missing tests, refactoring tests; no production code change)
- `chore:` (Updating grunt tasks etc; no production code change)

**Example:**
`feat: add new emotion recognition risk rule`

## Pull Request Process

1. Ensure any install or build dependencies are removed before the end of the layer when doing a build.
2. Update the README.md or relevant documentation with details of changes to the interface.
3. Test your changes locally to ensure they don't break existing functionality.
4. Push your branch to your fork on GitHub: `git push origin feat/your-feature-name`
5. Open a **Pull Request (PR)** against the `main` branch of the original repository.
6. Provide a clear and descriptive title for your PR, and explain exactly what changes you made and why.
7. A maintainer will review your PR. They may ask for changes or clarifications before merging.

## Reporting Bugs

If you find a bug, please create an Issue on GitHub. Provide as much information as possible, including:
- A clear, descriptive title.
- Steps to reproduce the bug.
- Expected behavior vs. actual behavior.
- Your environment details (OS, Node version, Python version, etc.).
